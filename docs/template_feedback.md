### 2. ⚠️ HIGH: Pre-Commit Config YAML Syntax Error

**File**: `.pre-commit-config.yaml`
**Line**: 66
**Severity**: High (breaks pre-commit hooks)

**Issue**:
```yaml
entry: bash -c 'command -v trufflehog >/dev/null 2>&1 && trufflehog git file://. --since-commit HEAD --results=verified,unknown --fail || echo "TruffleHog not installed - skipping (install: brew install trufflehog)"'
```

The single-quoted string contains unescaped single quotes in the parenthetical, causing YAML parsing to fail.

**Expected**:
```yaml
entry: bash -c 'command -v trufflehog >/dev/null 2>&1 && trufflehog git file://. --since-commit HEAD --results=verified,unknown --fail || echo "TruffleHog not installed - skipping"'
```

**Impact**:
- Pre-commit hooks fail to run
- `git commit` fails with YAML parsing error
- Workaround requires `--no-verify` (defeats security scanning)

**Context**:
Discovered when attempting to commit planning documents. Required manual fix before successful commit.

**Suggested Fix**:
1. Remove parenthetical from echo message OR escape properly
2. Add YAML linting to cookiecutter CI
3. Test pre-commit config in generated project during CI

### 3. 📝 MEDIUM: Front Matter Validation Hook Too Strict for Planning Documents

**File**: `.pre-commit-config.yaml`
**Hook**: `validate-front-matter`
**Severity**: Medium (blocks planning document commits)

**Issue**:
The front matter validation hook requires all markdown files to have strict YAML front matter matching specific schemas. However, planning documents generated by `/plan` command naturally don't have this front matter initially.

**Errors Encountered**:
```
docs/planning/project-vision.md: ISSUES
  - redundant H1 found: '# Project Vision & Scope...' — remove it; 'title' renders automatically
  - : Unable to extract tag using discriminator 'schema_type'

docs/planning/adr/adr-001-initial-architecture.md: ISSUES
  - redundant H1 found: '# ADR-001: ...' — remove it; 'title' renders automatically
  - : Unable to extract tag using discriminator 'schema_type'
```

**Impact**:
- Planning documents cannot be committed without `--no-verify`
- Defeats purpose of having planning document templates
- Inconsistent with `/plan` command workflow

**Context**:
All planning documents (project-vision.md, tech-spec.md, roadmap.md, ADRs) failed front matter validation even though they follow standard markdown structure.

**Suggested Fix Options**:

**Option 1** (Recommended): Exclude planning documents from strict front matter validation
```yaml
- id: validate-front-matter
  files: ^docs/(?!planning/).*\.md$  # Exclude docs/planning/
```

**Option 2**: Make planning document schemas optional
```python
# In validation script
if file_path.startswith('docs/planning/'):
    schema_required = False
```

**Option 3**: Auto-generate front matter in `/plan` command
- Update `.claude/skills/project-planning/` templates to include minimal front matter
- Ensure `schema_type: planning` is added to all generated docs

### 4. 📄 LOW: Documentation Clarity - Planning Document Workflow

**Severity**: Low (documentation improvement)

**Issue**:
The `docs/planning/` directory README states planning documents are "Awaiting Generation" but doesn't clearly explain the relationship between:
1. The `/plan` command (generates comprehensive documents)
2. The template placeholders (minimal scaffolds)
3. The front matter validation requirements

**Impact**:
- User confusion about which documents to generate vs. manually write
- Unclear whether to use `/plan` command or write manually
- Front matter validation failures seem unexpected

**Suggested Fix**:
Update `docs/planning/README.md` with clear workflow:

```markdown
## Generation Workflow

### Option 1: Claude Code `/plan` Command (Recommended)
1. Run `/plan <project description>` in Claude Code
2. Documents generated without front matter (validation excluded)
3. Comprehensive planning documents ready for implementation

### Option 2: Manual Creation
1. Copy templates from `.claude/skills/project-planning/templates/`
2. Add front matter header (required for validation)
3. Fill in project-specific content

**Note**: Planning documents generated by `/plan` are excluded from strict
front matter validation to support rapid iteration.
```

### 5. 🔧 LOW: ClusterFuzzLite Fails on Empty Project

**Job**: ClusterFuzzLite (address)
**Severity**: Low (expected for new project)

**Issue**:
ClusterFuzzLite fuzzing job fails on newly generated projects because there are no fuzz targets yet.

**Impact**:
- Red CI check on first PR (cosmetic, but confusing)
- Suggests CI failure when project is actually fine

**Suggested Fix**:
Add condition to skip fuzzing on empty projects:
```yaml
- name: Run ClusterFuzzLite
  if: hashFiles('fuzz/*.py') != ''
  # ... existing config
```

Or add placeholder fuzz target in template:
```python
# fuzz/example_fuzz.py
"""Placeholder fuzz target - replace with actual targets."""
def TestOneInput(data: bytes) -> None:
    pass
```
